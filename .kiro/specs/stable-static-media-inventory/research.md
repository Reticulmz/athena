- **Context**: Beatmap thumbnail / preview audio の取得、fallback、blob 保存、missing handling を設計するため。

- **Findings**:
  - `BeatmapFileProviderService` は `.osu` file を current official、legacy official、community mirrors の順で取得する。
  - Temporary failures は次の source へ fallback し、permanent errors は mirror fallback に進まない。
  - `FetchBeatmapFileUseCase` は idempotent fetch state を使い、expected md5 と fetched md5 を照合してから blob に保存し、attachment metadata を記録する。

  - Beatmap media 取得も source priority、fallback failure、idempotent fetch state、blob storage を同じ設計語彙で扱える。
  - Media-specific な stable route では、official/mirror の失敗を direct に漏らさず、404 と short negative cache に変換する。

## Route Inventory Matrix

### Matrix schema and allowed values

- Matrix rows represent one client-observable contract each. Split rows when method, host alias, path pattern, response shape, or redirect behavior differs.
- Required columns are the table columns below. Do not rename, remove, or reorder them without updating `design.md` `StableStaticMediaInventoryMatrix` and `Logical Data Model` together.
- Behavior columns are `response contract candidate`, `cache headers`, `content type`, `redirect`, `missing asset response`, and `expiry behavior`. If a behavior value lacks confirmed Stable Compatibility Evidence, record `needs-reference` rather than an unspecified placeholder.
- `compatibility classification` allowed values:
  - `required`: in-scope stable compatibility row required for a supported stable surface.
  - `deferred`: in-scope stable compatibility candidate kept for later implementation.
  - `out-of-scope`: adjacent or rejected surface not implemented by this spec.
  - `needs-reference`: row cannot be safely classified until target stable client traffic or equivalent evidence confirms the contract.
- `implementation priority` allowed values: `P1`, `P2`, `P3`.
  - Required for every `required` or `deferred` row.
  - Required for a `needs-reference` row when non-observable preparation work may proceed before final client-visible contract evidence. The priority records preparation order only, not implementation readiness for the observable behavior.
  - Leave empty only when `compatibility classification` is `out-of-scope` or when a `needs-reference` row cannot yet be associated with any in-scope preparation work.
- `current Athena coverage` allowed values:
  - `missing`: route is not registered, or current behavior does not cover the recorded contract.
  - `partial`: route path exists but one or more recorded behavior columns are not covered.
  - `implemented`: route path and all recorded behavior columns are covered by current Athena code.
- `evidence status` allowed values:
  - `confirmed`: target stable client traffic or equivalent Stable Compatibility Evidence confirms the row.
  - `needs-reference`: client-visible behavior still needs evidence before implementation-ready handoff.
  - `stable-reference-candidate`: docs or private-server behavior suggests the row, but target stable evidence has not confirmed it.
  - `rejected`: later Stable Compatibility Evidence disproves the candidate.
  - `superseded`: later Stable Compatibility Evidence replaces the candidate with a different row or behavior value.

### Evidence gate and implementation readiness

- `confirmed` は target stable client traffic、captured fixture、または同等の Stable Compatibility Evidence の source がある row だけに使う。`docs/stable-compatibility-guide.md` と `docs/stable-compatibility-matrix.md` は private-server / reference implementation 由来の Stable Reference Candidate source であり、それだけでは `confirmed` にしない。
- `evidence status=needs-reference`、`evidence status=stable-reference-candidate`、または behavior column のいずれかに `needs-reference` を含む row は、`final client-visible contract blocked by evidence gate` と扱う。Response body、status、header、content type、redirect、missing asset response、expiry、checksum source、host rewrite の最終 contract をこの row から確定実装してはならない。
- Evidence gate で blocked の row でも、`preparation-only allowed` と明記できる範囲は進められる。許されるのは fixture extraction 計画、row ownership の整理、内部 metadata / storage / cache boundary の設計など、client-visible contract を固定しない non-observable preparation work に限る。
- `implementation priority` が `P1` / `P2` / `P3` でも、未確認 row では準備順序だけを表す。Priority は final client-visible contract implementation readiness を意味しない。
- Stable Compatibility Evidence が Stable Reference Candidate と矛盾した場合は、target stable evidence を優先する。候補自体が誤りなら該当 row を `rejected` とし、別の row または behavior value に置き換わるなら旧 row を `superseded` として replacement row / source を `response contract candidate` または近接メモに記録する。
- `rejected` / `superseded` row は fixture extraction と review の監査履歴として残せるが、後続実装 scope にはしない。実装対象にするのは evidence source を持つ `confirmed` row、または observable contract を固定しない preparation-only task に限定する。

### Fixture extraction row id convention

- Every audited row must have a non-empty `fixture extraction row` value before handoff to #17 fixture extraction.
- Format: `stable-static-media.<family>.<method>.<host>.<path>.<response>.<redirect>`.
- Use lowercase ASCII kebab-case segments. Do not use spaces, raw URLs, query strings, or unstable database ids.
- Segment order is fixed: method, host alias, path pattern, response shape, redirect behavior.
- Normalize method as `get`, `post`, `head`, or the lowercase HTTP method token.
- Normalize host alias as `canonical` for the default `osu.$DOMAIN` or path-only route, or as the explicit host token such as `a-host`, `assets-host`, `b-host`, `d-host`, `d-osu-host`, `s-host`, `bare-domain`, or `ha-host`.
- Normalize path pattern to a compact route token such as `web-osu-screenshot`, `ss-id`, `ss-id-checksum`, `a-filename`, `forum-download-avatar`, `mt-filename`, `images-map-thumb`, or `mp3-preview`.
- Normalize response shape to a compact token such as `numeric-id`, `media-bytes`, `redirect`, `json`, `empty`, `missing`, or `adjacent-scope`.
- Normalize redirect behavior as `no-redirect` when the row does not redirect, or as a compact target/status token such as `to-ss-checksum-301`.
- Use `needs-reference` and `redirect-needs-reference` segments only when the row exists to preserve an unconfirmed candidate whose response or redirect shape must not be assumed.
- Handoff completion は audited matrix table が 91 rows、blank id 0、duplicate id 0、`stable-static-media` prefix、id ごとの dot-separated segment 数 7 を満たす状態とする。
- #17 readability のため、row id は隣接 row を読まなくても method、host alias、path pattern、response scenario、redirect behavior を特定できる必要がある。Method、host、path が同じ row は generic id を再利用せず、observable difference を response または redirect segment に入れる。
- Keep the id stable after fixture handoff. If #17 requires a different format, update this convention, the matrix rows, and `design.md` together.
- Examples: `stable-static-media.screenshot.post.canonical.web-osu-screenshot.numeric-id.redirect-needs-reference`, `stable-static-media.screenshot.get.canonical.ss-id.redirect.to-ss-checksum-301`, `stable-static-media.avatar.get.canonical.a-filename.media-bytes.no-redirect`, `stable-static-media.beatmap-thumbnail.get.b-host.thumb-filename.media-bytes.redirect-needs-reference`, `stable-static-media.preview-audio.get.canonical.mp3-preview.media-bytes.no-redirect`.

## Design Decisions

### Decision: Route matrix を research.md に分離する

- **Context**: Route ごとの behavior gap と evidence status は requirements より詳細で、後続の fixture extraction と design に再利用する必要がある。
- **Alternatives Considered**:
  1. `requirements.md` に matrix を置く
  2. `research.md` に matrix を置く
- **Selected Approach**: `research.md` に route inventory matrix を置き、requirements は満たすべき成果条件だけにする。
- **Rationale**: Requirements と evidence を分けることで、要件生成後も route candidate、未確認 gap、fixture extraction 対象を更新しやすい。
- **Trade-offs**: spec 読者は requirements と research の両方を見る必要がある。
- **Follow-up**: `/kiro-spec-requirements stable-static-media-inventory` の後、matrix rows を source docs と traffic evidence に基づいて埋める。

### Decision: needs-reference row は確定実装をブロックする

- **Context**: Host-Based Alias や screenshot upload などは Stable Reference Candidate から予測できるが、target stable client の exact request / response contract は未確認の可能性がある。
- **Alternatives Considered**:
  1. Stable Reference Candidate があれば仮実装して後で直す
  2. Client-observable contract が未確認の row は確定実装をブロックする
- **Selected Approach**: `needs-reference` row は確定実装をブロックする。ただし storage、metadata model、fixture extraction planning、内部 boundary 設計など、observable contract を固定しない non-observable preparation work は可能にする。
- **Rationale**: Stable compatibility は client-observable behavior が source of truth であり、推測実装は後で fixture と衝突しやすい。
- **Trade-offs**: static/media の実装開始は遅くなる。
- **Follow-up**: Static/media implementation は score submission、player online state / presence sharing、user stats display など Stable Gameplay Core Workflow の後に進める。

### Decision: Avatar serving は asset source の前提を含む

- **Context**: Athena は blob storage を使うため、avatar serving route だけを設計しても avatar blob と metadata の投入経路がなければ動作確認できない。
- **Alternatives Considered**:
  1. Avatar serving route だけをこの spec の範囲にする
  2. Avatar serving の範囲に Static/Media Asset Source の前提を含める
- **Selected Approach**: Avatar serving は blob asset が必要なため、Static/Media Asset Source の前提を scope に含める。ただし user-facing avatar upload/update workflow を同じ implementation scope に含めるかは別途判断する。
- **Rationale**: Serving contract と fixture extraction は返却対象の blob asset が存在して初めて検証できる。
- **Trade-offs**: Static/media scope が serving-only より少し広がる。
- **Follow-up**: Avatar asset を admin/dev seed、operator import、user-facing upload/update のどの経路で投入するかを決める。

### Decision: Avatar asset ingestion は共有 command use-case に集約する

- **Context**: Avatar serving は blob asset と metadata を必要とし、将来的には user-facing API と operator import の複数入口が avatar asset を投入する可能性がある。
- **Alternatives Considered**:
  1. User-facing API と operator import がそれぞれ別々に blob storage へ書き込む
  2. Avatar asset ingestion を共有 command use-case に集約し、各入口は同じ boundary を通す
- **Selected Approach**: Avatar asset ingestion は共有 command use-case として設計し、user-facing API と operator import は同じ validation、metadata update、blob write boundary を通る。
- **Rationale**: 入口ごとに画像検証、content metadata、avatar checksum、default/fallback behavior が分岐すると Stable Surface の serving contract が壊れやすい。
- **Trade-offs**: 初期の operator import だけを作る場合でも、use-case boundary を先に定義する必要がある。
- **Follow-up**: 実装 scope では API route と operator import route のどちらを先に公開するかを別途決める。

### Decision: Avatar variant は original 保存 + persistent cache にする

- **Context**: Reference candidate では `/a/<filename>` が `s=25|128|256` を受け付けるため、avatar serving は original blob だけではなく stable client-visible size variant を返す必要がある。
- **Alternatives Considered**:
  1. リクエストごとに original blob から resize して返す
  2. Upload/import 時にすべての variant を eager 生成する
  3. Original blob を保存し、初回要求または更新時に variant を生成して blob に永続キャッシュする
- **Selected Approach**: Original blob を保存し、`25` / `128` / `256` px の Avatar Serving Variant は lazy resize + persistent blob cache で扱う。
- **Rationale**: 毎 request の画像変換は CPU-bound で tail latency に影響しやすい。Variant storage cost は avatar サイズでは小さく、beatmap/replay assets と比べて相対的に低い。
- **Trade-offs**: Variant blob と metadata の追加 storage を消費する。
- **Follow-up**: Design phase で image processing library、variant content type、cache invalidation、checksum 更新タイミングを決める。Dependency 追加が必要な場合はユーザー承認を得る。

### Decision: Avatar serving variant は image/png に正規化する

- **Context**: Stable Reference Candidate では `/a/` と `/a/<filename>` が avatar bytes を `image/png` として返す。
- **Alternatives Considered**:
  1. Original の JPEG/PNG content type を serving variant でも維持する
  2. Stable client に返す Avatar Serving Variant を `image/png` に正規化する
- **Selected Approach**: Stable client に返す Avatar Serving Variant は `image/png` に正規化する。Original blob は再生成用に保持できる。
- **Rationale**: Stable Surface の content type が固定され、cache behavior と fixture expectation が安定する。
- **Trade-offs**: JPEG original から PNG variant を生成する場合、variant size が増える可能性がある。
- **Follow-up**: Design phase で alpha handling、background fill、maximum input size、invalid image rejection を決める。

### Decision: Avatar checksum source は serving variant hash を candidate にする

- **Context**: Stable responses may include filenames shaped like `<user_id>_<avatar_checksum>.png`, and avatar cache busting depends on the checksum source matching client expectations.
- **Alternatives Considered**:
  1. Original blob hash
  2. Stable serving variant content hash
- **Selected Approach**: Stable serving variant の content hash 由来を candidate とする。
- **Rationale**: Stable client が実際に取得する bytes と cache busting key が一致し、PNG 正規化や variant 再生成の影響を反映できる。
- **Trade-offs**: Reference implementations may expect a different checksum source.
- **Follow-up**: Target stable client traffic または同等 evidence で `<user_id>_<avatar_checksum>.png` の checksum source を確認するまでは `needs-reference` とする。

### Decision: Screenshot upload と serving は一つの compatibility workflow として扱う

- **Context**: Screenshot は stable client の upload route で blob asset を投入し、その response id を serving route が受ける。
- **Alternatives Considered**:
  1. Screenshot upload と serving を別 route family として個別に扱う
  2. Screenshot Compatibility Workflow として upload、blob ingestion、id response、expiry、hidden/missing behavior、serving headers を一続きに扱う
- **Selected Approach**: Screenshot は Screenshot Compatibility Workflow として扱う。
- **Rationale**: Upload response contract と serving route contract が同じ screenshot id と blob metadata に依存するため、分けると fixture extraction と implementation tasks がずれやすい。
- **Trade-offs**: Screenshot scope は serving-only より広くなる。
- **Follow-up**: Stable client の upload auth query、multipart field name、accepted image types、size limit、expiry checksum source を evidence で確認する。

### Decision: Screenshot は original content type のまま serving する

- **Context**: Reference candidate は screenshot serving を PNG/JPEG image bytes として扱い、avatar のような size variant 正規化を要求していない。
- **Alternatives Considered**:
  1. Screenshot を `image/png` に正規化する
  2. Upload された JPEG/PNG の original content type を保持して serving する
- **Selected Approach**: Screenshot upload は JPEG/PNG を受け付け、original content type のまま保持・serving する。
- **Rationale**: Screenshot はリサイズや stable avatar variant と異なり、client が送った画像 asset をそのまま id で取得する compatibility workflow である。
- **Trade-offs**: Serving route は screenshot ごとに content type が変わる。
- **Follow-up**: Accepted upload filename/type、max size、MIME sniffing policy は evidence と design phase で確定する。

### Decision: Screenshot expiry は configurable default unlimited にする

- **Context**: Reference candidate には seven-day expiry があるが、Athena では screenshot を期限切れで消すことを基本動作にしない方針。
- **Alternatives Considered**:
  1. Reference candidate に合わせて default seven-day expiry にする
  2. Screenshot Expiry Policy を configurable にし、default は unlimited にする
- **Selected Approach**: Screenshot Expiry Policy は configurable とし、Athena の default は unlimited にする。
- **Rationale**: Screenshot は operator policy で保存期間を変えたい可能性があり、default expiry でユーザー生成 asset が消える挙動は private server 運用では不要な驚きになりやすい。
- **Trade-offs**: Reference candidate の expiry behavior とは異なる可能性がある。
- **Follow-up**: Seven-day expiry を期待する stable client behavior があるかは `needs-reference` として確認する。

### Decision: Screenshot serving checksum は md5 candidate にする

- **Context**: Reference candidate は `/ss/<id>` から checksum 付き URL への redirect を示しているが、checksum source が target stable client で重要かは未確認である。
- **Alternatives Considered**:
  1. Created-at md5
  2. Content hash 由来の md5
  3. Stable client が送る checksum value
- **Selected Approach**: Screenshot Serving Checksum は md5 形式を candidate とし、source value は `needs-reference` として確認する。
- **Rationale**: Stable client や reference route が md5-shaped value を期待する可能性を残しつつ、Athena の blob source of truth に合わせた content hash への余地も残せる。
- **Trade-offs**: Source value が確定するまで serving redirect contract は実装ブロックになる。
- **Follow-up**: Target stable client traffic で `/ss/<id>`、`/ss/<id>/<checksum>`、upload response の関係を確認する。

### Decision: Beatmap media は公式 source 優先で取得し blob cache から配信する

- **Context**: Beatmap thumbnail と preview audio routes は stable client-visible cache headers、content type、missing asset behavior、host alias と結びつくため、upstream source の HTTP behavior を直接露出したくない。
- **Alternatives Considered**:
  1. Upstream source へ redirect する
  2. Upstream source を proxy して response をそのまま返す
  3. 公式 source を第一候補に取得し、取得失敗や unavailable の場合は mirror source へ fallback して、Athena blob に保存して配信する
- **Selected Approach**: Beatmap thumbnails と preview audio は Beatmap Media Source Priority に従い、公式 source を第一候補、mirror source を fallback として取得する。取得した media は Beatmap Media Mirror Cache として blob に保存して Athena から配信する。
- **Rationale**: Stable Surface の cache headers、content type、404 behavior、host alias を Athena 側で統一できる。
- **Trade-offs**: Media blob storage、source priority、refresh policy、fallback failure handling が必要になる。
- **Follow-up**: Design phase で official source の利用条件、fallback source、freshness、negative cache、missing upstream asset、content type validation、thumbnail/preview key を決める。

### Decision: Beatmap media fallback failure は 404 + short negative cache にする

- **Context**: Official source と mirror source の両方から beatmap thumbnail / preview audio を取得できない場合、stable client-visible route behavior を決める必要がある。
- **Alternatives Considered**:
  1. Upstream failure をそのまま proxy する
  2. 毎 request で upstream retry する
  3. Stable route は 404 を返し、Beatmap Media Negative Cache で短時間 upstream retry を抑制する
- **Selected Approach**: Stable route は missing asset response として 404 を返し、Beatmap Media Negative Cache を短時間持つ。
- **Rationale**: Stable client には単純な missing asset として見せつつ、unavailable upstream への繰り返し問い合わせで latency と外部負荷が増えることを避ける。
- **Trade-offs**: Negative cache TTL 中に upstream asset が復旧しても即時反映されない。
- **Follow-up**: Design phase で TTL、permanent/temporary failure category、manual refresh、observability fields を決める。

### Decision: Menu / seasonal / title image は classification と priority を分けて残す

- **Context**: Menu / seasonal / title image routes は core login/play を止める可能性は低いが、stable client が実際に叩く Static/Media Route Inventory の候補である。
- **Alternatives Considered**:
  1. Out-of-scope として matrix から外す
  2. classification `deferred` かつ priority `P3` として matrix に残す
- **Selected Approach**: Menu / seasonal / title image は classification `deferred` かつ priority `P3` として matrix に残す。
- **Rationale**: 実装優先度が低くても、Stable Surface の候補なら route classification と evidence gap を見える状態にしておくべきである。
- **Trade-offs**: Matrix rows が増える。
- **Follow-up**: Target stable client traffic で実利用が確認されなければ、後続 design/tasks で実装をさらに延期できる。

## Risks & Mitigations

- Route family と path pattern を混同するリスク — Fixture Extraction Row を client-observable contract 単位に分ける。
- Stable Reference Candidate を confirmed contract と誤読するリスク — Evidence status を必須列にする。
- Host-Based Alias の exact path を推測で実装するリスク — 初期監査では `needs-reference` として traffic evidence を要求する。

### Implementation Approach Options

#### Option A: Existing stable web legacy を拡張する

- 内容: `application.py` に routes/hosts を追加し、`transports/stable/web_legacy` と `StableWebLegacyProviderSet` に handlers を増やす。Blob storage と beatmap fetch service を直接再利用する。
- 良い点: 既存の stable PHP-compatible endpoint pattern に近く、初期の route registration は速い。
- 悪い点: screenshot、avatar、beatmap media、menu の責務が web_legacy に集中しやすい。metadata、variant generation、negative cache まで transport 側に引き寄せると layer boundary が崩れる。
- 向く範囲: inventory-only route stub、薄い transport adapter、既存 route style の確認。

#### Option B: Static/media 専用 bounded feature を新設する

- 内容: `domain/static_media` または既存 domain context 下に asset language を置き、commands/queries/repositories/transports を専用に切る。
- 良い点: screenshot workflow、avatar variants、beatmap media cache、menu assets の責任境界が明確になる。fixture extraction と serving contract をテストしやすい。
- 悪い点: 初期ファイル数と schema/interface 設計が増える。beatmap fetch や blob storage と似た語彙を重複させない設計が必要。
- 向く範囲: P1/P2 implementation、metadata persistence、operator import/API 共通 ingestion。

#### Option C: Hybrid

- 内容: static/media の domain/service/repository boundary は専用に作り、Starlette routing、stable web legacy adapter style、BlobStorageService、beatmap source priority/fetch-state pattern は再利用する。
- 良い点: transport は薄く保ちつつ、既存の blob/beatmap infrastructure を活かせる。route matrix と fixture rows から task を分割しやすい。
- 悪い点: 設計フェーズで route family ごとの ownership と migration order を明確にしないと、Option A と B の中途半端な混在になる。
- 推奨度: 現時点では最も妥当な候補。特に avatar ingestion を API と operator import の共通 command use-case にする方針と相性がよい。

### Design Phase Recommendations

- Preferred approach candidate: Option C。transport registration は既存 stable web legacy pattern に合わせ、observable behavior 以外の storage/source/variant/cache は dedicated static/media services に置く。
- Route matrix は design の前半で必ず実 row に展開する。初期 row は `current Athena coverage=missing` とし、docs 由来は `Stable Reference Candidate`、traffic 未確認は `needs-reference` にする。
- P1 を先に設計する場合も、screenshot uploadとserving、avatar sourceとserving、beatmap media fetchとservingの依存関係を分けて task 化する。
- Blob storage はそのまま再利用できるが、screenshot/avatar/beatmap media の attachment metadata は各 feature ownership で定義する。polymorphic attachment は roadmap の blob-storage 方針に反するため避ける。
- Avatar resize は original保存 + lazy persistent variant cache を前提候補にし、画像処理ライブラリは design phase で候補比較と承認を取る。
- Beatmap media は既存 `.osu` fetch の official-first/fallback/idempotent fetch-state を参照する。ただし media serving は upstream HTTP behavior を露出せず、Athena-controlled 404/cache/content-type に変換する。
- Local blob backend は水平スケール上の制約がある。production design では S3-compatible backend 前提の読み書き境界を崩さない。

## Design Discovery And Synthesis

### Discovery Classification

- **Feature Type**: Extension
- **Discovery Process**: Light discovery
- **Reason**: 既存 stable compatibility docs と runtime route coverage を監査する brownfield extension であり、新しい runtime service、DB schema、外部 dependency の導入はこの spec の所有範囲外である。

### Integration-Focused Discovery

- **Context**: `$kiro-spec-design stable-static-media-inventory -y` により requirements を承認して design draft を生成するため、gap analysis の後に design boundary を確定する。
- **Sources Consulted**:
  - `.kiro/specs/stable-static-media-inventory/requirements.md`
  - `.kiro/specs/stable-static-media-inventory/research.md`
  - `docs/stable-compatibility-guide.md`

  - `src/osu_server/composition/application.py`

  - `.kiro/steering/tech.md`
  - `.kiro/steering/roadmap.md`
  - `.kiro/steering/scaling.md`
- **Findings**:
  - Current runtime routing covers `c.$DOMAIN`, `c<int>.$DOMAIN`, `ce.$DOMAIN`, and `osu.$DOMAIN`, but not `a.$DOMAIN`, `assets.$DOMAIN`, `b.$DOMAIN`, `d.$DOMAIN`, `d.osu.$DOMAIN`, `s.$DOMAIN`, bare domain, or `ha.$DOMAIN`.
  - Current `osu.$DOMAIN` stable web routes cover registration, bancho connect, getscores, and score submit only. Static/media routes are not registered.
  - `docs/stable-compatibility-guide.md` and `docs/stable-compatibility-matrix.md` already contain reference candidates for screenshots, avatars, beatmap media, menu assets, and host aliases, but they do not separate confirmed evidence from reference candidates at row granularity.
  - Blob storage, beatmap file fetch, multipart parser, and rate limiter are reusable future implementation assets, but this inventory spec does not need to modify them.
- **Implications**:
  - The design should treat `research.md` route inventory matrix as the owned artifact.
  - Runtime route registration, blob attachment tables, image processing, and media serving services should be follow-up implementation scopes, not hidden work inside this inventory spec.

### Design Synthesis

#### Generalization

All audited static/media surfaces can be modeled as one `Stable Static/Media Inventory Row`. Screenshot upload, screenshot serving, avatar serving, beatmap thumbnail, preview audio, menu assets, and host aliases differ by row attributes rather than by separate documentation formats. The row schema therefore becomes the shared contract for design, fixture extraction, and later implementation tasks.

#### Build vs Adopt

No new runtime library or external tool is adopted in this spec. The inventory is maintained as Markdown in `research.md` because the required output is a human-reviewable compatibility matrix, not a runtime data store. Future avatar resizing or media transcoding libraries remain unselected and require a later dependency approval.

#### Simplification

The design does not introduce code, database tables, generated schemas, or service interfaces for runtime serving. It records future implementation boundary recommendations only where needed to keep follow-up tasks from guessing ownership.

### Decision: この spec は inventory artifact を所有し runtime serving を実装しない

- **Context**: Requirements は static/media route inventory、classification、evidence gate、current coverage、fixture extraction row を求める。一方で actual serving implementation は Stable Gameplay Core Workflow の後で進める方針である。
- **Alternatives Considered**:
  1. この spec で runtime static/media services と routes まで設計する
  2. この spec は inventory artifact と follow-up readiness を所有し、runtime implementation は別 spec/task へ分離する
- **Selected Approach**: この spec は route inventory matrix、evidence gate、coverage audit、follow-up handoff を所有する。Runtime routes、DB schema、blob attachments、image processing dependency は後続 spec で設計する。
- **Rationale**: Requirements の主目的は推測に依存しない route inventory であり、runtime 実装まで含めると未確認 evidence と Stable Gameplay Core Workflow の優先順位に反する。
- **Trade-offs**: Design だけでは static/media routes は実装されない。
- **Follow-up**: Design approval 後の tasks は `research.md` matrix completion、docs cross-check、fixture extraction handoff に集中する。
