# Implementation Plan

- [ ] 1. CI ワークフロー作成

- [x] 1.1 品質チェックジョブの構築
  - PR 作成・更新および main プッシュ時に自動起動するワークフローを新規作成する
  - 同一ブランチの古い実行を自動キャンセルする concurrency 設定を追加する
  - uv によるパッケージインストールとキャッシュを設定し、ジョブ末尾に cache prune を追加してキャッシュ肥大化を抑制する
  - basedpyright が CI 環境で正しく venv を参照できるようにする（devenv 固有パスへのシンボリックリンク作成）
  - ruff lint、ruff format 準拠、basedpyright 型チェック、import-linter をそれぞれ独立したステップとして実行する
  - いずれかのステップが失敗しても残りのステップが実行を継続する設定を行う
  - ワークフローを push し、GitHub Actions 上で quality ジョブが起動し全4ステップの成否が個別に表示されることを確認する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 3.1, 4.2, 4.3_

- [x] 1.2 テストジョブの構築
  - pytest を実行するテストジョブを品質チェックジョブと並列で動作するように追加する
  - テスト実行に必要な PostgreSQL と Redis のサービスコンテナを health check 付きで設定する
  - データベースと Redis への接続情報を環境変数で提供する
  - pytest が失敗した場合、テスト名とトレースバックが GitHub Actions のログ上で確認できることを検証する
  - テストジョブが正常に起動し、pytest がサービスコンテナに接続してテストを実行できることを確認する
  - _Requirements: 2.1, 2.2, 3.1, 4.1_
  - 1.1 と同一ファイルを編集するため並列実行不可

- [ ] 2. ブランチ保護の設定
  - main ブランチに対して、品質チェックジョブとテストジョブの両方が成功しない限りマージを禁止するブランチ保護ルールを設定する
  - ステータスチェック名は CI ワークフローが1回以上実行された後に GitHub に登録される。初回 CI 実行後に保護ルールを適用する
  - ステータスチェック失敗時に PR のマージボタンが無効化されていることを確認する
  - _Depends: 1.1, 1.2_
  - _Requirements: 5.1, 5.2_
  - _Blocked: プライベートリポジトリでは GitHub Pro が必要。public 化またはアップグレード後に設定可能_

- [ ] 3. CI パイプラインの統合検証
  - PR 作成、PR 更新（追加コミット）、main 直接プッシュの3パターンでワークフロートリガーを検証する
  - quality ジョブ内で意図的にチェック違反を含むコミットを push し、失敗したステップ以外が正常に実行されることを確認する
  - 全チェック成功時にマージ可能、いずれか失敗時にマージ不可であることを確認する
  - _Depends: 2_
  - _Requirements: 1.5, 3.1, 4.1, 4.2, 4.3, 5.2_
  - _Blocked: Task 2 がブロック中のため統合検証不可_
