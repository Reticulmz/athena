# ADR 0010: Store API Tokens as Hashed Credentials

## Status
Accepted (2026-06-17)

## Context
Athena will allow users or integrations to issue API tokens for Public API access. These tokens are long-lived credentials, so storing raw bearer tokens would make the database and backups a credential store with high breach impact. Web App API routes use cookie sessions, CSRF protection, and Web App Sudo Mode instead of API token authentication.

## Decision
Athena shows the raw API token only once at issue time and never stores it durably. The raw token is a structured string containing token type, key_id, and a server-generated CSPRNG 256-bit-or-higher URL-safe secret. PostgreSQL stores only a keyed token hash / HMAC value, hash_key_version, display_prefix, key_id, scope, expires_at, last_used_at, and owner/status metadata needed for listing, authorization, usage tracking, expiration, and revocation. HMAC keys are not stored in PostgreSQL; they are supplied as a versioned runtime key ring from AppConfig, environment variables, or a future secret manager. HMAC key rotation uses a verify-many / issue-one model: new tokens are issued with the active key, while existing tokens keep verifying with the key selected by their stored hash_key_version. Public API token authentication accepts only `Authorization: Bearer <token>` and rejects query parameter tokens. Idempotency replay of an API token issue workflow must not redisplay the raw token or secret; it returns token metadata and indicates that the secret cannot be shown again. Initial scopes are coarse `read`, `write`, and `admin`; endpoint-specific scopes are deferred until the public API and operational workflows are stable enough to justify finer-grained authorization.

## Consequences
API token issue and delete workflows require Web App Sudo Mode and produce Security/Audit Events. Token lookup narrows candidates using the non-secret key_id parsed from the raw token and then verifies the keyed hash / HMAC with the runtime key selected by hash_key_version, while UI listing uses only non-secret metadata such as display_prefix, scope, expiry, status, and last-used time. If a caller loses the raw token after issue time, recovery is revoke and reissue rather than redisplay or in-place regeneration. Old HMAC keys stay available for verification until tokens using that hash_key_version expire or are rotated. When an old key must be retired, affected tokens enter forced rotation with Web App UI and API metadata warnings; after the retirement deadline, authentication fails with `token_key_retired`. Token validity failures return HTTP 401 and use codes such as `invalid_token`, `expired_token`, `revoked_token`, and `token_key_retired`; valid tokens without enough scope return HTTP 403 with `insufficient_scope`. This keeps integration branching machine-readable without encoding token state differences primarily in HTTP status, while still separating authentication failure from authorization failure. Scope checks should stay conservative and coarse initially rather than creating premature endpoint-level policy debt. Web App API `/api/web/*` must not accept API tokens as an authentication mechanism.
