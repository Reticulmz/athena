# ADR 0011: Store Public API Idempotency Records Durably

## Status
Accepted (2026-06-17)

## Context
Public API mutations may include high-risk operations such as billing integration, moderation changes, and API token issue or revoke workflows. These operations need safe retry behavior without allowing restart, failover, or cache eviction to reopen the same mutation for duplicate side effects.

## Decision
Athena stores Public API Idempotency Records in PostgreSQL, not Valkey alone. Idempotency Records are created only after API token authentication succeeds and the owner/token identity is known; unauthenticated or invalid-token failures are handled by the normal Public API failure, rate limit, and audit paths instead of idempotency storage. The idempotency key, owner/token identity, HTTP method, route, request fingerprint, and outcome are finalized in the same transaction or Unit of Work consistency boundary as the target mutation. The request fingerprint is a hash derived from canonicalized method, route, relevant query, and body intent; raw request bodies, API tokens, passwords, and payment payloads are not stored in the idempotency record. Replayable outcomes are successful mutation results and deterministic 4xx validation or authorization failures. 5xx responses and timeouts are not stored as replayable failures, so callers may retry transient server-side failures; if a durable mutation has already committed, the committed success outcome remains the idempotency outcome and prevents duplicate side effects. A retry with the same key and same request fingerprint returns the original HTTP status, response body, and contract response headers after completion, except that one-time secrets such as newly issued raw API tokens are not replayed and are represented only by non-secret metadata plus a "not displayable again" state. Replayable headers include result headers such as `Location`, `ETag`, `Sunset`, and `Deprecation`; request-specific or hop-by-hop values such as `Date`, `Server`, `Set-Cookie`, and `request_id` are generated for the new request context instead of being replayed. If the original request is still in progress, Athena does not hold the second request open; it returns HTTP 409 with `idempotency_request_in_progress` and `Retry-After`. The same key with a different payload or intent returns HTTP 409 with `idempotency_key_conflict`. The initial retention default is 7 days, configurable through operational settings. After retention expires, the record may be pruned and the original deduplication guarantee no longer applies.

## Consequences
Valkey may still be used as a lookup cache or write-through optimization, but PostgreSQL remains the source of truth for high-risk Public API idempotency. This adds durable writes to selected mutation paths, but it prevents duplicate billing, moderation, and credential-management side effects across process restarts and infrastructure failover without turning the idempotency table into a raw request payload archive. Not recording transient server failures preserves recovery by retry, while recording deterministic client-side failures keeps repeated invalid requests from drifting into different outcomes under the same key. A pruning job is required so idempotency records do not grow without bound.
