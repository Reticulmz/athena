# ADR 0008: Use Server-Side Session Cookies for Web App Authentication

## Status
Accepted (2026-06-17)

## Context
Athena Web App is a first-party browser application served with same-origin Web App API routes. Authentication could use bearer tokens in browser storage, stateless JWTs, or backend-owned server-side sessions.

## Decision
Athena Web App authentication will use FastAPI-issued server-side sessions referenced from an HttpOnly, Secure, SameSite=Lax cookie. Active Web App Session state is stored as TTL-based Valkey state, while PostgreSQL remains the source of truth for user state, authorization source data, audit records, and security events. Next.js does not own the authentication source of truth and must not store bearer tokens in localStorage.

## Consequences
Web App and Web App API must remain same-origin by default so cookies can be used without cross-origin credential complexity. FastAPI owns login, logout, session rotation, expiration, and authorization refresh for Web App Sessions. A Web App Session has an idle timeout, an absolute lifetime, and session ID rotation on login, privilege escalation, and sensitive-operation boundaries. Initial configurable defaults are idle timeout 12 hours, absolute lifetime 30 days, and Web App Sudo Mode window 15 minutes. Web App Sudo Mode is required for password/email changes, 2FA setup, moderation mutations, Admin/Ops mutations, billing operations, and API token issue/delete workflows, but not for normal browsing or low-risk profile edits. API token raw values are shown only once at issue time; durable storage keeps only token hash, non-secret prefix, scope, expires_at, last_used_at, and owner/status metadata. Destructive or high-impact moderation and Admin/Ops operations additionally require Operator Intent Confirmation so the operator explicitly confirms the target, action, and reason where required. Security-sensitive workflows and operator actions are stored as PostgreSQL append-only Security/Audit Events containing actor, target, operation, reason, request/session context, and outcome. Security/Audit Events must cover both successful operations and failed attempts such as authorization failure, Web App Sudo Mode failure, CSRF failure, and Operator Intent Confirmation failure. Security/Audit Events must not store secrets, raw tokens, passwords, payment card data, or full request payloads. The initial configurable retention default is one year, and expired records are removed by a retention pruning job rather than ad hoc deletion. Forced logout, privilege changes, and verification changes are reflected through session authorization refresh or versioning rather than by treating browser cookies as the authority. Next.js may read only the derived authenticated state needed for rendering and routing, while protected mutations continue through FastAPI command/query use-cases.
